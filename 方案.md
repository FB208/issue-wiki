# 核心目标

开发一个面向 GitHub 开源项目的任务管理、赞助、共创评论、说明文档综合平台。

平台第一版只服务一个 GitHub 项目，但项目地址通过 `.env` 配置，方便部署到不同开源项目。平台本身也作为开源项目维护。

第一版先不做 GitHub 深度集成，例如 GitHub 登录、自动同步 issue、展示 PR、展示版本、展示贡献者等；后续版本可以在现有用户、任务、文档模型上扩展。

# 第一版范围

第一版包含：

- 游客浏览、点赞、赞助。
- 用户注册、登录、提需求、共创评论、查看自己的需求和赞助记录。
- 管理员管理任务、文档、用户、评论、订单。
- 前台和后台均适配桌面端与移动端。
- 任务赞助支付接入爱发电，使用备注/留言中的功能 ID 回填到具体任务。
- Markdown 编辑与实时预览。
- 文件上传到 RustFS 图床/文件存储。
- GitHub Actions 根据版本 tag 自动构建镜像并部署。

第一版不包含：

- GitHub 登录。
- GitHub issue 自动同步。
- 公开赞助榜、赞助人列表、赞助留言展示。
- 文档版本历史、回滚、删除恢复。
- 退款。
- 审计日志。
- 数据和附件备份策略。
- 项目内处理域名、HTTPS、Nginx 反向代理。

# 技术架构

- 后端：Python FastAPI。
- 前端：React，前台和后台使用同一个 React 应用。
- 数据库：MySQL。
- 缓存/验证码限制：Redis。
- 数据库迁移：Alembic。
- Markdown：统一封装轻量级 Markdown 编辑器，支持实时预览。
- 文件存储：RustFS，连接配置写入 `.env`。
- 容器：Docker，前后端打包为一个应用镜像。
- 部署：GitHub Actions + Docker Registry + SSH 部署到 Linux 服务器。

# 前端风格

整体采用 `开源控制台` 风格：简洁、有高级感，接近成熟 SaaS 控制台，强调清晰的信息层级、任务表格、筛选条件、数据卡片和后台管理效率。

桌面端采用左右结构的标准 Wiki/控制台布局：左侧是固定导航区，右侧是内容区。

移动端必须完整适配，不允许只做桌面端缩放。移动端要求：

- 左侧导航在移动端收起为抽屉菜单或顶部菜单入口。
- 任务表格在移动端改为卡片列表，保留顺序、标题、启动资金、已赞助金额、共创人数、状态和主要操作。
- 筛选条件在移动端收起为筛选面板，支持任务名称搜索和状态多选。
- 赞助、提需求、登录、评论等核心操作在移动端可直接完成。
- Markdown 编辑器在移动端可正常编辑、预览、上传图片和附件。
- 文档阅读在移动端优先保证正文可读性，目录可收起。
- 管理后台在移动端可完成常用管理操作，包括任务审核、评论处理、订单查看和用户封禁。
- 表单、弹窗、支付页需要适配手机屏幕，避免横向滚动和按钮不可点击。
- 桌面端、平板端、手机端都需要作为开发验收范围。

- 可以浏览赞助功能和公开文档。
- 可以查看任务列表、任务详情、文档详情。
- 可以给任务赞助。
- 可以给文档点赞。
- 不可以提需求。
- 不可以共创评论任务。
- 不可以评论文档。

## 普通用户

- 拥有游客能力。
- 可以提需求。
- 可以查看自己创建的需求。
- 可以在未完成任务下共创评论。
- 可以评论文档。
- 可以查看自己的赞助记录。

## 管理员

- 通过 `/admin` 进入后台。
- 管理员账号通过 `.env` 配置管理员邮箱和密码。
- 后台与前台属于同一个 React 应用。
- 管理员可以管理任务、文档、用户、评论、订单。
- 不需要后台操作审计日志。

# 左侧菜单

- 固定菜单：`赞助功能`。
- `赞助功能` 是首页，进入系统后默认打开。
- 其他菜单来自管理员创建的文档和文件夹。
- 文档和文件夹支持层级结构。
- 文档和文件夹支持自定义排序。
- 文档支持在不同文件夹之间移动。

# 赞助功能

## 任务状态

任务状态包括：

- 待审核。
- 待启动。
- 进行中。
- 已完成。

说明：

- 用户提需求创建的任务默认是 `待审核`。
- 管理员审核通过后进入 `待启动`。
- 启动资金只是展示给用户看的募集目标金额，不自动改变任务状态。
- 已赞助金额达到启动资金后，也不自动改变任务状态。
- 隐藏任务使用独立的可见性字段处理，不作为任务状态。

## 管理员任务管理

管理员可以录入和维护任务。

任务字段包括：

- 名称。
- 描述。
- 顺序。
- 创建日期。
- 修改日期。
- 启动资金。
- 任务状态。
- 是否隐藏。

系统计算字段包括：

- 已赞助金额：根据已支付赞助订单自动汇总，不允许手工录入。
- 共创人数：按任务下参与共创评论的去重用户数计算，只统计评论用户。

任务描述使用统一 Markdown 编辑器，支持实时预览和附件上传。

任务管理能力：

- 新增任务。
- 编辑任务。
- 隐藏任务。
- 审核用户提交的需求。
- 调整任务状态。
- 拖拽调整任务顺序。

任务顺序规则：

- 顺序必须唯一。
- 管理员可以拖拽排序。

查询条件：

- 任务名称模糊查询。
- 任务状态下拉多选。

排序字段：

- 顺序。
- 名称。
- 启动资金。
- 已赞助金额。

## 用户提需求

仅登录用户可以提需求。

用户提需求时可以填写：

- 名称。
- 详细描述。

用户提需求时不允许填写：

- 启动资金。
- 顺序。
- 任务状态。

提交后的任务默认状态是 `待审核`，需要管理员审核通过后才进入 `待启动` 并出现在默认任务列表中。

已登录用户可以查看自己创建的需求，包括待审核需求。

## 任务显示

游客和普通用户可以查看任务列表。

默认只显示：

- 进行中任务。
- 待启动任务。

默认排序：

- 按顺序从小到大排列。

展示效果：

- 前三个任务需要设计金、银、铜牌视觉效果。

表格字段：

- 顺序。
- 标题。
- 创建日期。
- 启动资金。
- 已赞助金额。
- 共创人数。

查询条件：

- 任务名称模糊查询。
- 任务状态下拉多选。

排序字段：

- 顺序。
- 名称。
- 启动资金。
- 已赞助金额。

页面操作：

- 提需求。
- 共创。
- 赞助。

## 任务共创

任务共创本质上是任务评论。

规则：

- 仅登录用户可以共创。
- 已完成任务不能继续共创。
- 未完成任务可以继续共创。
- 共创内容使用统一 Markdown 编辑器。
- 共创内容支持上传图片和附件。
- 单个上传文件大小限制为 10MB。

管理员评论管理：

- 评论不需要提前审核。
- 管理员可以确认评论。
- 管理员可以回复评论。
- 管理员可以删除评论。

## 任务赞助

游客和登录用户都可以赞助任务。

赞助流程：

- 用户点击 `赞助`。
- 弹出赞助说明，展示当前功能 ID，例如 `IW-TASK-12`。
- 用户复制功能 ID，跳转到爱发电自选金额赞助。
- 用户在爱发电备注/留言中填写功能 ID。
- 爱发电 webhook 回调成功订单后，系统生成已支付赞助订单。
- 如果备注中有有效功能 ID，订单计入对应任务；未填写或无法匹配时，订单记为“赞助作者”，不增加到具体任务。
- 任务已赞助金额根据已支付且已绑定任务的订单汇总。

赞助规则：

- 赞助金额由爱发电页面决定。
- 爱发电回调订单无法可靠映射到本项目登录用户，统一记录为游客赞助。
- 任务完成后仍然可以继续赞助。
- 不支持退款。

登录用户赞助记录：

- 爱发电回调订单暂不映射到本项目用户，登录用户个人赞助记录后续接入爱发电 OAuth 后再完善。

后台订单管理：

- 管理员可以查看订单列表。
- 管理员可以按任务、用户、订单状态、创建时间查询订单。
- 管理员不能手工把未支付订单改成已支付，支付成功以爱发电 webhook 回调为准。

支付接入要求：

- 使用爱发电开发者文档：https://guide.afdian.com/creator/developer
- 内部订单需要记录爱发电订单号、爱发电用户标识、方案 ID、备注、金额、支付状态、回调时间。
- 订单状态和字段映射以爱发电文档为准，`status=2` 表示交易成功。
- webhook 地址使用 secret 查询参数保护。
- webhook 必须按爱发电订单号做幂等处理，重复通知不能重复累加赞助金额。
- 备注中无法解析功能 ID、任务不存在或格式无效时，订单记为“赞助作者”。

# 文档功能

文档由管理员创建，游客和普通用户只能查看。

文档结构：

- 支持文件夹层级。
- 支持文档在文件夹之间移动。
- 支持文件夹和文档自定义排序。

文档字段：

- 标题。
- 内容。
- 创建时间。
- 修改时间。
- 作者，默认是 `生产力Mark`。

文档编辑：

- 内容使用统一 Markdown 编辑器。
- 支持实时预览。
- 支持图片和附件上传。
- 管理员保存后立即公开。
- 不需要草稿状态。
- 不需要版本历史。
- 不需要回滚。
- 不需要删除恢复。

文档互动：

- 登录用户可以点赞文档。
- 登录用户可以评论文档。
- 游客可以点赞文档。
- 游客不能评论文档。
- 游客点赞不做严格防刷限制。

管理员评论管理：

- 文档评论不需要提前审核。
- 管理员可以确认评论。
- 管理员可以回复评论。
- 管理员可以删除评论。

# Markdown 与附件

系统封装统一 Markdown 编辑器，任务描述、任务共创、文档内容共用同一套能力。

基础能力：

- Markdown 编辑。
- 实时预览。
- 图片上传。
- 附件上传。
- 轻量级 Markdown 渲染。

上传规则：

- 文件上传到 RustFS。
- RustFS 连接信息配置在 `.env`。
- 单个文件大小限制为 10MB。
- 支持图片、PDF、Word、Excel、Markdown 文件。
- 需要限制文件扩展名和 MIME 类型。

Markdown 增强能力：

- 优先选择轻量级编辑器和渲染工具。
- 代码高亮、目录锚点、任务列表、表格、Mermaid 等能力不自行实现。
- 如果所选工具原生支持，可以直接启用；第一版不为这些能力做重型定制。

# 注册和登录

## 注册

注册字段：

- 邮箱或手机号，二选一。
- 昵称。
- 密码。
- 头像，可选。

注册验证：

- 手机号注册必须使用阿里云短信验证码。
- 邮箱注册必须使用邮箱验证码。
- 邮箱验证码通过 Gmail SMTP 发送。
- Gmail SMTP 只是发信服务，允许所有邮箱注册。

短信接口文档：

- https://help.aliyun.com/zh/pnvs/developer-reference/api-dypnsapi-2017-05-25-sendsmsverifycode?spm=5176.11662647.console-base_help.dexternal.f24a5a79XsR8JU

## 登录

登录方式：

- 邮箱或手机号 + 密码。

登录规则：

- 仅支持密码登录。
- 注册时提示用户仅支持密码登录，需要牢记密码。
- 登录状态长期不过期。
- 密码必须哈希存储，不能明文保存。

## 忘记密码

找回方式：

- 注册使用邮箱，则通过邮箱验证码找回。
- 注册使用手机号，则通过短信验证码找回。

验证码限制：

- 每日每个邮箱或手机号最多发送 2 次验证码。
- 发送次数限制使用 Redis 记录。

# 部署方案

## Docker

- 前端和后端打包成一个 Docker 镜像。
- 容器对外暴露一个网址。
- MySQL 不打包进应用镜像。
- Redis 不打包进应用镜像。
- RustFS 不打包进应用镜像。
- MySQL、Redis、RustFS 通过 `.env` 配置连接。

## GitHub Actions

触发规则：

- 当提交 tag 符合 `vX.X.X` 格式时触发自动部署。

部署流程参考根目录 `deploy-backend.yml`，但需要适配本项目：

- Checkout 代码。
- 从 tag 提取版本号。
- 设置 Docker Buildx。
- 登录 Docker Registry。
- 构建前后端一体应用镜像。
- 推送版本镜像和 `latest` 镜像。
- 通过 SSH 登录 Linux 服务器。
- 在服务器部署目录执行 `docker compose pull`。
- 执行 `docker compose up -d --no-build`。
- 清理旧镜像。
- 发送部署结果邮件。

需要的 GitHub Secrets：

- `REGISTRY`。
- `REGISTRY_USERNAME`。
- `REGISTRY_PASSWORD`。
- `REGISTRY_REPO`。
- `SSH_HOST`。
- `SSH_USER`。
- `SSH_PORT`。
- `SSH_PASSWORD`。
- `DEPLOY_DIR`。
- `GMAIL_USERNAME`。
- `GMAIL_PASSWORD`。
- `GMAIL_TO`。

## 数据库迁移

- 使用 Alembic 管理数据库结构变更。
- 部署新版应用时需要执行数据库迁移。
- 迁移失败时不应继续启动新版本应用。

## 服务器边界

- 域名不在项目内处理。
- HTTPS 不在项目内处理。
- Nginx 反向代理不在项目内处理。
- 数据库备份不在项目内处理。
- 附件备份不在项目内处理。
